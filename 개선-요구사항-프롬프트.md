## 에이전트 응답 (원문)

아래는 **tizenclaw-cli를 실제로 쓰면서 아쉽다고 느낄 수 있는 점들**을 기준으로,
**UX / 성능 / 기능 / 에러 처리 / 운영성 / 확장성**까지 전반적으로 정리한 개선 제안입니다.

---

### 1. 전체 총평

tizenclaw-cli는 단순 질의응답을 넘어서 작업 디렉터리 기반 파일 작업, 코딩 에이전트
위임, 세션/메모리 활용, 도구 호출 기반 실행 같은 강력한 기능을 제공하는 형태입니다.

대표적 문제:

- **무엇을 할 수 있는지 직관적으로 알기 어렵다**
- **현재 무슨 일이 일어나고 있는지 투명성이 부족하다**
- **에러가 났을 때 사용자가 복구하기 어렵다**
- **반복 작업을 자동화하기엔 CLI ergonomics가 부족하다**
- **긴 작업/에이전트 위임/파일 생성 작업에서 피드백이 약하면 답답하다**
- **메모리/세션/작업공간 개념이 강력하지만 오히려 복잡하게 느껴질 수 있다**

핵심: 기능 자체보다 **사용자 제어감, 예측 가능성, 디버깅 가능성**을 강화해야 한다.

---

### 2. UX 측면의 문제점

#### 2-1. 학습 곡선이 높다

작업 디렉터리, 세션, 장기 메모리, 코딩 에이전트 위임, 도구 사용, 파일 생성/수정 등의
개념이 함께 들어가 초기 사용자가 혼란을 겪기 쉽다.

개선 방향:
- `tizenclaw --help` 구조화 (기본 사용 / 파일 작업 / 코딩 작업 / 세션·메모리 / 자동화)
- `tizenclaw doctor` 또는 `tizenclaw explain-context`: 현재 작업 디렉터리, 활성 백엔드,
  메모리 사용 여부, 사용 가능한 도구, 파일 쓰기 가능 범위
- 첫 실행 시 `quickstart` 제공

#### 2-2. 현재 상태가 잘 보이지 않는다

CLI 기반 에이전트에서 가장 큰 UX 불만은 "지금 뭘 하고 있는지 모르겠다"는 점이다.

개선 방향:
- 스트리밍 상태 표시 강화: `Thinking…`, `Reading files…`, `Delegating to coding agent…`
- 단계형 실행 로그 옵션: 기본 간단 상태 → `--verbose` 세부 단계 → `--debug` 메타데이터
- 장기 작업에는 진행률 또는 heartbeat ("30초째 실행 중")

#### 2-3. 명령 결과의 예측 가능성이 부족하다

같은 프롬프트에 CLI가 바로 답할지, 파일을 만들지, 도구를 쓸지, 에이전트에 위임할지
예측하기 어렵다.

개선 방향:
- `--mode answer | tool | plan | safe`
- `--dry-run`: 실제 실행 전에 "이 요청에 대해 다음 행동이 예상됩니다" 출력
- 파일 변경/삭제/이동은 사전 요약 제공

#### 2-4. CLI 출력물이 사람이 읽기에 최적화되지 않았다

긴 응답, 코드 블록, 상태 로그, 최종 결과가 뒤섞이면 읽기 어렵다.

개선 방향:
- 출력 레이어 분리: 상태 로그는 stderr, 최종 답변은 stdout
- `--output text|json|markdown`
- `--quiet`: 최종 답변만 출력

---

### 3. 성능 측면의 문제점

#### 3-1. 긴 작업에서 지연 체감이 크다

개선 방향:
- 병렬 가능한 작업은 병렬화
- 짧은 요청에는 경량 경로 제공 (단순 질답은 무거운 컨텍스트 로딩 최소화)
- 캐시 도입: 최근 문서 조회, 동일 디렉터리 파일 인덱스, 자주 쓰는 시스템 상태

#### 3-2. 컨텍스트 로딩 비용이 크다

개선 방향:
- 필요한 컨텍스트만 lazy loading
- 메모리 요약과 상세를 분리 (기본은 summary만)
- 세션 컨텍스트 상한선 시각화 ("현재 컨텍스트 사용량: 68%")

#### 3-3. 무거운 코딩 에이전트 위임의 비용이 크다

개선 방향:
- 위임 판단 로직 개선: 단순 파일 편집/짧은 요약은 직접 처리, 다단계 코드 변경은 위임
- `--no-delegate`, `--delegate codex`, `--delegate-if-complex` 옵션
- 위임 기준 설명

---

### 4. 기능 누락 또는 강화 필요 영역

#### 4-1. 대화형 CLI 기능 부족

개선 방향:
- REPL 모드: `tizenclaw chat`
- 세션 내 명령: `/status`, `/memory`, `/files`, `/undo`, `/plan`
- 멀티턴 작업 히스토리 탐색

#### 4-2. 패치/수정 워크플로우

개선 방향:
- diff-first 모드: 변경 전 실제 패치 미리보기
- `--apply` / `--preview`
- 자동 백업 또는 undo stack
- 파일별 변경 이유 요약

#### 4-3. 프로젝트 탐색 기능

개선 방향:
- `tizenclaw index`: 프로젝트 구조 요약, 주요 엔트리 포인트 추정
- `tizenclaw find "keyword"`, `tizenclaw summarize dir/`
- 언어별 코드맵 생성

#### 4-4. 반복 작업 자동화

개선 방향:
- 구조화 출력 강화를 위한 `--json`
- exit code 표준화
- 비대화형 실행 최적화
- 템플릿 명령 (release note, commit summary, PR 리뷰 요약, 로그 분석)

#### 4-5. 메모리 관리 기능

개선 방향:
- `tizenclaw memory list`, `show <id>`, `delete <id>`, `disable`
- 응답에 메모리 반영 여부 표시 옵션

---

### 5. 에러 처리 측면의 문제점

#### 5-1. 에러 메시지가 기술적이거나 모호하다

에러 메시지를 3단 구성으로 개선:
1. 무엇이 실패했는지
2. 왜 그럴 가능성이 큰지
3. 사용자가 무엇을 하면 되는지

예시:
- 나쁜 예: `tool call failed`
- 좋은 예: "파일 읽기에 실패했습니다. 원인: 지정한 경로가 작업 디렉터리 밖이거나 파일이
  존재하지 않을 수 있습니다. 해결: `pwd`, `ls`, 또는 절대 경로를 확인해 주세요."

#### 5-2. 복구 가능한 에러와 치명적 에러 구분이 약하다

개선 방향:
- recoverable error 분류: 파일 없음 → 대체 경로 제안, 문서 검색 실패 → 재시도 제안
- fatal error 분류: 권한 없음, 쓰기 금지 위치, 백엔드 미설정

#### 5-3. timeout / 네트워크 실패 대응이 아쉽다

개선 방향:
- 자동 재시도 정책
- 부분 결과 보존
- `--timeout` 사용자 설정
- 장기 작업 resume 지원

#### 5-4. destructive action 보호가 더 필요하다

개선 방향:
- 위험 작업은 기본 확인, `--force`로만 무확인 실행
- trash/backup 전략, 수정 전 스냅샷 저장

---

### 6. 기능적 신뢰성 측면의 문제

#### 6-1. "성공했다"는 말과 실제 결과가 다를 수 있다는 불안

개선 방향:
- 완료 검증 루틴 강화: 파일 생성 후 크기/형식 확인, 비어 있는 출력 방지
- 최종 보고 형식 표준화 (생성 파일 / 수정 파일 / 검증 결과 / 남은 리스크)

#### 6-2. 부분 성공(partial success)을 제대로 다뤄야 한다

개선 방향:
- 상태를 4단계로 구분: Success / Partial Success / Failed / Blocked
- 부분 성공 시 세부 항목 명시

---

### 7. 개발자 관점의 개선 포인트

- `--debug-trace`: request id, tool call id, 실행 단계별 structured log, 실패 재현용 스냅샷
- 테스트 가능한 인터페이스: golden test용 deterministic mode, mockable tool layer
- 설정 관리: 우선순위 체계(CLI flag > env > project config > global), `tizenclaw config
  show/edit`, 프로파일(default / fast / safe / coding)

---

### 8. 협업/실무 관점의 개선 포인트

- 실행 결과를 Markdown/JSON 리포트로 저장
- 세션 내보내기, 공유 가능한 run recipe
- 프로젝트별 정책 파일 (허용 도구, 기본 출력 형식, 위험 작업 승인 정책)
- CI/CD 친화성: non-interactive safe mode, 명확한 exit codes, stdout/stderr 역할 분리

---

### 9. 문서화 측면의 문제점

문서를 세 층으로 분리:
1. 빠른 시작
2. 작업별 레시피 (파일 수정, 프로젝트 요약, 보고서 생성, 코딩 에이전트 위임, JSON 출력 연결)
3. 고급 참조

실패 사례 중심 문서 추가 ("왜 파일이 안 만들어졌나요?", "왜 응답이 오래 걸리나요?" 등)

---

### 10. 우선순위별 개선 제안

#### 1순위 (꼭 먼저 개선)

1. 상태 가시성 강화: 지금 무엇을 하고 있는지 표시
2. 에러 메시지 개선: 원인 + 해결 방법 포함
3. 파일 변경 안전장치: preview/diff/undo
4. 출력 모드 정리: human-readable / json / quiet
5. 위임 판단의 투명성: 왜 위임했는지 설명

#### 2순위 (실사용 만족도를 크게 올리는 것)

1. REPL/chat 모드
2. 메모리 관리 명령
3. 프로젝트 탐색/index 기능
4. timeout/retry/resume 개선
5. config/profile 관리

#### 3순위 (장기적 경쟁력)

1. CI/automation 친화성
2. 팀 공유/세션 export
3. deterministic test mode
4. 정책 기반 실행 제어
5. 풍부한 템플릿/워크플로우 제공

---

### 11. 이상적인 tizenclaw-cli의 모습

개선이 잘 되면 tizenclaw-cli는:

- **설명 가능한 에이전트 CLI**
- **안전하게 파일 작업하는 자동화 도구**
- **프로젝트 탐색과 코딩 작업을 돕는 실무형 인터페이스**
- **개인/팀/CI 환경 모두에서 일관되게 쓰이는 플랫폼**

핵심 4가지: **예측 가능성, 투명성, 복구 가능성, 자동화 적합성**

---

### 12. 한 줄 결론

현재 tizenclaw-cli의 가장 큰 아쉬움은 "강력한 기능에 비해 사용자가 시스템의 상태와
의도를 충분히 통제·이해하기 어렵다"는 점이고, 가장 중요한 개선 방향은 **상태 표시,
안전한 파일 변경, 명확한 에러 처리, 구조화된 출력, 메모리/위임 제어 강화**이다.
