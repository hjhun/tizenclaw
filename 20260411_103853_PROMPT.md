# TizenClaw Follow-up Prompt

## Context

The current cycle aligned TizenClaw with the reference projects in
three areas:

- runtime capability visibility for shell/interpreter/file utility paths
- linux-utility-first file operations with Rust fallbacks
- embedded descriptor assessment with migration guidance toward textual
  skills or built-in runtime features

## Remaining High-Value Work

1. Port OpenClaw-style shell trust and wrapper analysis into the tool
   execution path so shell wrappers, inline command carriers, and skill
   preludes are auditable before execution.
2. Separate tool-pool assembly from skill-pool assembly more cleanly,
   following OpenClaude's model, so prompt injection only receives the
   minimal tool and skill inventory required for the turn.
3. Add a first-class skill capability manager inspired by Hermes Agent:
   dependency checks, disabled-skill configuration, external skill root
   reporting, and install guidance surfaced through IPC and the CLI.
4. Revisit embedded descriptors one by one and decide whether each item
   should remain built in, move to a textual skill, or be removed.
5. Add focused system scenarios for `file_manager` operations so linux
   utility execution and Rust fallback behavior remain observable in
   host-first regression runs.

## Constraints

- Keep host-first routing through `./deploy_host.sh`
- Preserve script-driven verification and avoid direct ad-hoc cargo
  commands
- Keep documents in English and operator communication in Korean
- Prefer Clean Architecture boundaries and TDD-first slices
