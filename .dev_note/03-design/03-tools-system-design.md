## Design Specification

### 1. Unified Tool Context Injection (ToolWatcher & PromptBuilder)
The `ToolWatcher` (`src/core/tool_watcher.rs`) is already implemented as a fast Tokio interval task watching via `std::fs::metadata`. When directories under `/opt/usr/share/tizen-tools/` change, the trigger fires a callback.
**Design Adjustment for Prompt Integration:**
- The Core logical loop (`agent_core.rs` or `context_engine.rs`) must bind `ToolWatcher::set_change_callback` to a state modification channel (e.g. `tokio::sync::mpsc::unbounded_channel()` or simply wrapping an `Arc<RwLock<SystemPrompt>>`).
- When the callback is invoked, the agent rebuilds `available_skills` and tool index Markdown reading directly from `/opt/usr/share/tizen-tools/skills/` (dynamically listing `SKILL.md` equivalent files).
- `SystemPromptBuilder` is then invoked with exactly these updated `skills`, mapping directly into the next API inference request without a daemon restart.

### 2. ActionBridge Seamless JSON Parsing (v1/v2 schema resilience)
Instead of rigidly demanding specific Tizen Action Framework keys, `ActionBridge::sync_action_schemas_from` (`src/core/action_bridge.rs`) will implement associative abstraction.
- We continue to deserialize into `serde_json::Value`.
- Key resolution will perform heuristic lookups:
  - `action_id`: Fallback to `id`, `name`.
  - `display_name`: Fallback to `title`, `name`.
  - `description`: Default to display name if absent.
  - `parameters`: Default to `Value::Null`.
  - `app_id`: Fallback to `package_id` or `""`.
- Since there is no concrete `struct` defining the deserialization, `serde_json::from_str::<Value>` provides a zero-cost wrapper abstracting the unpredictable Tizen Action Native runtime without `panic!()` risks inside the persistent daemon loop.

### 3. Removal of Custom Skill References
- The `SystemPromptBuilder` tests and generation logics map solely to `<available_skills>` without mentioning any `custom_skills`.
- `agent_roles.json` structures will remove the custom array fields, shifting to a single monolithic skill iteration.
- `tizenclaw_secure_container.sh` container bindings will map `/opt/usr/share/tizen-tools/skills` strictly inside the OCI configuration parameters.

### Execution Boundaries
- **Send + Sync:** `ToolWatcher` callback utilizes `Box<dyn Fn() + Send + 'static>`, allowing safe memory updates to shared `RwLock` Contexts across the async logical engine.
- **FFI Boundary:** None introduced. All operations heavily rely on POSIX IO paths and `serde_json` memory mapping. The `ActionBridge` operates natively reading standard JSON exports generated by the real Tizen Action subsystem, completely decoupling our daemon from brittle C-API struct dependencies. `libloading` dynamic resolution is maintained in existing scopes without further modification here.

Completion of this stage dictates readiness for Phase 4 Code Development implementing the heuristic fallback parsers and cleaning `tools.md` indexes natively.
